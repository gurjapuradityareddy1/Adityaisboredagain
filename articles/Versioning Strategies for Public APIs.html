Versioning Strategies for Public APIs

Public APIs are contracts. Once released, breaking them breaks trust. Versioning is how APIs evolve without breaking consumers.

When Should You Version?

  • Breaking response changes
  • Removed fields
  • Behavioral changes

URL Versioning


GET /api/v1/orders
GET /api/v2/orders

Simple and explicit, but increases surface area.

Header-Based Versioning


Accept: application/vnd.company.v2+json
Preferred for mature platforms with many versions.

Backward-Compatible Changes

  • Add new fields (never remove)
  • Make fields optional

Deprecation Strategy


Deprecation: true
Sunset: 2026-01-01

Best Practices

  • Version only when necessary
  • Document breaking changes clearly
  • Give migration timelines
