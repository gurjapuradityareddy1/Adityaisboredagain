Error Handling in Distributed Systems

In distributed systems, failures are not exceptions â€” they are expected behavior. Unlike monolithic systems, a failure in one service can cascade across the system if not handled correctly.

Why Error Handling Is Hard in Distributed Systems

  • Network calls are unreliable
  • Services fail independently
  • Latency varies across hops
  • Partial failures are common
A system that does not plan for failure will fail unpredictably.

Types of Errors

  • Transient errors (timeouts, temporary outages)
  • Permanent errors (invalid requests)
  • Downstream dependency failures

Retry With Backoff

Retries should be controlled and time-bound.


function retry(fn, retries = 3) {
  let delay = 100;
  for (let i = 0; i < retries; i++) {
    try {
      return fn();
    } catch (e) {
      sleep(delay);
      delay *= 2;
    }
  }
  throw new Error("Retries exhausted");
}
Unbounded retries amplify outages and cause retry storms.

Circuit Breaker Pattern

Circuit breakers prevent repeated calls to failing services. 


if (failureRate > threshold) {
  circuit.open();
}

Graceful Degradation

When a dependency fails, return partial or cached data instead of failing completely.

Key Takeaways

  • Failures must be isolated
  • Retries must be bounded
  • Systems must fail predictably
