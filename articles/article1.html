API Documentation Template REST

Published: January 2026

Overview

This document is a ready-to-use template for writing clean and professional API documentation. You can reuse it for internal APIs, public developer portals, SDK guides, or integration docs.

Base URL

https://api.example.com/v1

Authentication

This API uses Bearer Token authentication.

Authorization: Bearer YOUR_API_KEY
Note: Keep API keys secret. Never expose them in frontend code or public repositories.

Request Format

  • Content-Type: application/json
  • Accept: application/json

Common Headers

Header Required Description
Authorization Yes Bearer token used to authenticate API requests.
Content-Type Yes (POST/PUT/PATCH) Must be application/json for JSON payload requests.
X-Request-Id No Optional request tracking ID for debugging and tracing.

Rate Limits

By default, the API supports 100 requests/minute per API key. If the limit is exceeded, the API returns 429 Too Many Requests.

Error Handling

All errors follow a consistent JSON format.

Error Response Example

{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "The 'email' field is required.",
    "requestId": "b7a1c8c9-2f0a-4d9c-8c3f-123456789abc"
  }
}

Common Status Codes

Status Code Meaning When It Happens
200 OK Request completed successfully.
201 Created A new resource was created successfully.
400 Bad Request Validation failed or request payload is invalid.
401 Unauthorized Missing or invalid authentication token.
403 Forbidden Token is valid but lacks required permissions.
404 Not Found The requested resource does not exist.
429 Too Many Requests Rate limit exceeded.
500 Internal Server Error Unexpected error on the server.

Endpoints

Below are sample endpoints that demonstrate the recommended documentation structure. Replace these with your real endpoints.

1) Create User

Method: POST

Endpoint: /users

Description: Creates a new user record.

Request Body

Field Type Required Description
name string Yes User's full name.
email string Yes User's email address (must be unique).
role string No User role (example: admin, viewer).

Request Example

POST /v1/users
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json

{
  "name": "John Doe",
  "email": "john@example.com",
  "role": "viewer"
}

Success Response

{
  "id": "usr_12345",
  "name": "John Doe",
  "email": "john@example.com",
  "role": "viewer",
  "createdAt": "2026-01-01T10:20:30Z"
}

2) Get User by ID

Method: GET

Endpoint: /users/{userId}

Description: Retrieves a user by unique ID.

Path Parameters

Parameter Type Required Description
userId string Yes The unique identifier of the user.

Request Example

GET /v1/users/usr_12345
Authorization: Bearer YOUR_API_KEY
Accept: application/json

Success Response

{
  "id": "usr_12345",
  "name": "John Doe",
  "email": "john@example.com",
  "role": "viewer",
  "createdAt": "2026-01-01T10:20:30Z"
}

3) List Users

Method: GET

Endpoint: /users

Description: Returns a paginated list of users.

Query Parameters

Parameter Type Required Description
page number No Page number (default: 1).
limit number No Number of results per page (default: 10).
role string No Filter users by role.

Request Example

GET /v1/users?page=1&limit=10
Authorization: Bearer YOUR_API_KEY

Success Response

{
  "page": 1,
  "limit": 10,
  "total": 42,
  "data": [
    {
      "id": "usr_12345",
      "name": "John Doe",
      "email": "john@example.com",
      "role": "viewer"
    },
    {
      "id": "usr_67890",
      "name": "Jane Smith",
      "email": "jane@example.com",
      "role": "admin"
    }
  ]
}

4) Update User

Method: PATCH

Endpoint: /users/{userId}

Description: Updates one or more fields of an existing user.

Request Example

PATCH /v1/users/usr_12345
Authorization: Bearer YOUR_API_KEY
Content-Type: application/json

{
  "role": "admin"
}

Success Response

{
  "id": "usr_12345",
  "name": "John Doe",
  "email": "john@example.com",
  "role": "admin",
  "updatedAt": "2026-01-10T12:00:00Z"
}

5) Delete User

Method: DELETE

Endpoint: /users/{userId}

Description: Deletes a user permanently.

Warning: This action is irreversible. Use carefully in production environments.

Request Example

DELETE /v1/users/usr_12345
Authorization: Bearer YOUR_API_KEY

Success Response

{
  "message": "User deleted successfully."
}
